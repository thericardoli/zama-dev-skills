# 测试 ERC7984 confidential token

本文件讲如何用 `forge-fhevm` 写 ERC7984 测试：把 `FhevmTest` 的输入、解密和 proof helper 用对，再围绕 token 行为做断言。

下面的接口名按 ERC7984 常见形态书写。项目如果封装了 mint、burn、disclose、wrapper 或 RWA 逻辑，把示例里的业务函数替换成项目自己的函数。

## 测试面

| 行为 | 用到的 forge-fhevm 能力 | 重点断言 |
| --- | --- | --- |
| balance / total supply | `userDecrypt` | 有权用户能解密，第三方不能 |
| mint / burn | `encryptUint64` + `userDecrypt` | 权限、余额、total supply 正确 |
| transfer | `encryptUint64` + `userDecrypt` | 余额更新，余额不足时 fail-closed |
| operator / transferFrom | `encryptUint64` | 未授权失败，授权成功，过期失败 |
| existing handle transfer | `userDecrypt` / ACL | 不能拿别人的 `euint64` handle 花钱 |
| transfer event amount | `vm.recordLogs` + `userDecrypt` | sender/recipient 的事件金额权限正确 |
| disclose / public reveal | `buildDecryptionProof` | proof ABI 编码和合约验证一致 |
| wrapper | `dealConfidential` | 测试前准备 wrapper confidential balance |

metadata、decimals、ERC165 也要测，但它们和 FHE 测试框架关系不大，按普通 Foundry 断言即可。

## 最小骨架

```solidity
import {FhevmTest} from "forge-fhevm/FhevmTest.sol";
import {euint64, externalEuint64} from "@fhevm/solidity/lib/FHE.sol";
import {IERC7984} from "@openzeppelin/confidential-contracts/interfaces/IERC7984.sol";

contract MyERC7984Test is FhevmTest {
    uint256 internal constant HOLDER_PK = 0xA11CE;
    uint256 internal constant RECIPIENT_PK = 0xB0B;
    uint256 internal constant OPERATOR_PK = 0x0DE;

    MyConfidentialToken internal token;

    address internal holder;
    address internal recipient;
    address internal operator;

    function setUp() public override {
        super.setUp();

        holder = vm.addr(HOLDER_PK);
        recipient = vm.addr(RECIPIENT_PK);
        operator = vm.addr(OPERATOR_PK);

        token = new MyConfidentialToken(/* args */);
        _mintForTest(holder, 1000);
    }
}
```

如果项目没有公开 mint/burn 接口，可以在 `test/` 下建 test-only harness 暴露必要 hook；不要把 harness 放进生产合约。

## 推荐 helper

```solidity
function _encryptAmount(uint256 pk, uint64 amount)
    internal
    returns (externalEuint64 encryptedAmount, bytes memory proof)
{
    return encryptUint64(amount, vm.addr(pk), address(token));
}

function _decryptHandle(uint256 pk, euint64 value) internal returns (uint64) {
    address user = vm.addr(pk);
    bytes memory sig = signUserDecrypt(pk, address(token));
    return uint64(userDecrypt(euint64.unwrap(value), user, address(token), sig));
}

function _decryptBalance(uint256 pk, address account) internal returns (uint64) {
    return _decryptHandle(pk, token.confidentialBalanceOf(account));
}

function _decryptTotalSupply(uint256 pk) internal returns (uint64) {
    return _decryptHandle(pk, token.confidentialTotalSupply());
}

function callUserDecrypt(bytes32 handle, address user, address contractAddress, bytes memory sig)
    external
    returns (uint256)
{
    return userDecrypt(handle, user, contractAddress, sig);
}
```

`callUserDecrypt` 只用于 `expectRevert`，因为 internal call 的 revert 不方便直接捕获。

加密输入的两条绑定最容易写错：

- `encryptUint64(amount, signer, address(token))` 的 target 必须是最终调用 `FHE.fromExternal` 的 token 合约。
- `signer` 应该和实际发起该次 token 操作的 caller/operator 一致。

## Balance 和 ACL

正向：余额 owner 能解密。

```solidity
function test_holderCanDecryptBalance() public {
    assertEq(_decryptBalance(HOLDER_PK, holder), 1000);
}
```

反向：第三方不能解密 holder 的 balance handle。

```solidity
function test_thirdPartyCannotDecryptBalance() public {
    uint256 thirdPk = 0xCAFE;
    address third = vm.addr(thirdPk);
    euint64 balance = token.confidentialBalanceOf(holder);
    bytes memory sig = signUserDecrypt(thirdPk, address(token));

    vm.expectRevert(
        abi.encodeWithSelector(
            FhevmTest.UserNotAuthorizedForDecrypt.selector,
            euint64.unwrap(balance),
            third
        )
    );
    this.callUserDecrypt(euint64.unwrap(balance), third, address(token), sig);
}
```

`confidentialTotalSupply()` 也按同样方式测：谁应该能解密就写正向，谁不该能解密就写反向。

## Mint / burn

mint/burn 使用同一套 encrypted input 和 decrypt helper：

```solidity
function test_mintUpdatesBalanceAndSupply() public {
    (externalEuint64 amount, bytes memory proof) = _encryptAmount(HOLDER_PK, 250);

    vm.prank(holder);
    token.mintConfidential(recipient, amount, proof); // 替换成项目自己的 mint 接口

    assertEq(_decryptBalance(RECIPIENT_PK, recipient), 250);
    assertEq(_decryptTotalSupply(HOLDER_PK), 1250);
}
```

同时覆盖：非 minter、zero address、burn 余额不足、burn 后 balance/total supply 是否符合设计。

## Transfer

外部输入版本：

```solidity
function test_transferEncryptedAmount() public {
    (externalEuint64 amount, bytes memory proof) = _encryptAmount(HOLDER_PK, 400);

    vm.prank(holder);
    token.confidentialTransfer(recipient, amount, proof);

    assertEq(_decryptBalance(HOLDER_PK, holder), 600);
    assertEq(_decryptBalance(RECIPIENT_PK, recipient), 400);
}
```

至少再覆盖这些失败路径：

- 余额不足：交易可以不 revert，但余额应保持不变。
- zero receiver：按项目/标准预期 revert。
- signer 和 caller 不一致。
- proof target 不是 token 合约。

## Operator / transferFrom

operator 的 encrypted input 应由 operator 签名，因为实际调用 `confidentialTransferFrom` 的人是 operator。

```solidity
function test_operatorTransferFrom() public {
    vm.prank(holder);
    token.setOperator(operator, uint48(block.timestamp + 1 days));

    (externalEuint64 amount, bytes memory proof) = _encryptAmount(OPERATOR_PK, 250);

    vm.prank(operator);
    token.confidentialTransferFrom(holder, recipient, amount, proof);

    assertEq(_decryptBalance(HOLDER_PK, holder), 750);
    assertEq(_decryptBalance(RECIPIENT_PK, recipient), 250);
}
```

同时测：

- 未授权 operator 调用失败。
- `vm.warp(expiry + 1)` 后 `isOperator(holder, operator)` 为 false，transferFrom 失败。
- operator 用 existing handle 时，holder 必须已经把该 handle 授权给 operator。

## Existing handle

如果 token 支持 `confidentialTransfer(to, euint64 amount)`，测试核心是 ACL，而不是 proof：

```solidity
function test_transferExistingHandle() public {
    euint64 holderBalance = token.confidentialBalanceOf(holder);

    vm.prank(holder);
    token.confidentialTransfer(recipient, holderBalance);

    assertEq(_decryptBalance(RECIPIENT_PK, recipient), 1000);
}
```

反向 case：`holder` 不能拿 `recipient` 的 balance handle 当 amount 转走；operator 也不能绕过 handle ACL。

## transferAndCall

如果项目支持 callback，保留一组行为测试即可：

- callback 成功：转账生效，接收合约拿到预期 data。
- callback 返回失败或拒收：按实现预期 revert 或 refund。
- callback 自身 revert：错误向上传递。
- 接收方是 EOA：按标准/项目预期处理。
- operator 走 `confidentialTransferFromAndCall` 时仍遵守 operator 授权和 input signer 绑定。

## Event amount

如果 transfer event 暴露 encrypted amount handle，用 `vm.recordLogs()` 抓事件，再对事件里的 `euint64` handle 做 user decrypt：

```solidity
vm.recordLogs();
_transferForTest(holder, recipient, 400);

euint64 eventAmount = _amountHandleFromTransferLog(vm.getRecordedLogs());
assertEq(_decryptHandle(HOLDER_PK, eventAmount), 400);
assertEq(_decryptHandle(RECIPIENT_PK, eventAmount), 400);
```

第三方解密 event amount 应失败。事件解析函数按项目事件结构写，不需要在 skill 里固定实现。

## Disclose / public reveal

如果项目支持公开 reveal 某个 encrypted amount，重点测 proof 编码：

```solidity
function test_discloseBalance() public {
    euint64 balance = token.confidentialBalanceOf(holder);

    vm.prank(holder);
    token.requestDiscloseEncryptedAmount(balance);

    uint64 clear = _decryptHandle(HOLDER_PK, balance);
    bytes memory proof = buildDecryptionProof(euint64.unwrap(balance), abi.encode(clear));

    vm.prank(holder);
    token.discloseEncryptedAmount(balance, clear, proof);
}
```

`publicDecrypt(handles)` 生成的 proof 默认匹配 `abi.encode(uint256[] cleartexts)`。如果合约验证的是 `abi.encode(uint64 clear)` 或其他自定义 ABI，使用 `buildDecryptionProof(handle, abi.encode(...))`。

失败路径至少测：无权 request、空 proof、错 handle、错 cleartext、重复 disclose。

## Wrapper

只有测试 `IERC7984ERC20Wrapper` 时才需要：

```solidity
dealConfidential(wrapper, user, amount);
```

它用于在测试前准备 wrapper 的 confidential balance。普通 ERC7984 token 不需要这个 helper。

## 常见错误

- input proof 的 target 不是 token 合约。
- encrypted input 的 signer 和 `msg.sender`/operator 不一致。
- 只给了 transient ACL，导致跨交易 `userDecrypt` 失败。
- 忘了给 balance、event amount 或 total supply 配置预期的 persistent ACL。
- operator 授权时间过期或 `vm.warp` 后未重新授权。
- disclose 使用了错误的 proof ABI 编码。
- 余额不足路径以为一定 revert；很多 confidential token 会 fail-closed，应该断言余额不变。
