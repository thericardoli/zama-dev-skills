# Testing ERC7984 Confidential Tokens

This file explains how to write ERC7984 tests with `forge-fhevm`: use the `FhevmTest` input, decrypt, and proof helpers correctly, then assert token behavior around them.

The interface names below follow common ERC7984 shapes. If the project wraps mint, burn, disclose, wrapper, or RWA logic, replace the example business functions with the project's own functions.

## Test Surface

| Behavior | forge-fhevm capability | Key assertions |
| --- | --- | --- |
| balance / total supply | `userDecrypt` | Authorized users can decrypt; third parties cannot |
| mint / burn | `encryptUint64` + `userDecrypt` | Permissions, balances, and total supply are correct |
| transfer | `encryptUint64` + `userDecrypt` | Balances update, and insufficient balance fails closed |
| operator / transferFrom | `encryptUint64` | Unauthorized calls fail, authorized calls succeed, expired operators fail |
| existing handle transfer | `userDecrypt` / ACL | A user cannot spend someone else's `euint64` handle |
| transfer event amount | `vm.recordLogs` + `userDecrypt` | Sender/recipient event amount permissions are correct |
| disclose / public reveal | `buildDecryptionProof` | Proof ABI encoding matches contract verification |
| wrapper | `dealConfidential` | Wrapper confidential balance is prepared before the test |

Metadata, decimals, and ERC165 should also be tested, but they are ordinary Foundry assertions and are not specific to the FHE test framework.

## Minimal Skeleton

```solidity
import {FhevmTest} from "forge-fhevm/FhevmTest.sol";
import {euint64, externalEuint64} from "@fhevm/solidity/lib/FHE.sol";
import {IERC7984} from "@openzeppelin/confidential-contracts/interfaces/IERC7984.sol";

contract MyERC7984Test is FhevmTest {
    uint256 internal constant HOLDER_PK = 0xA11CE;
    uint256 internal constant RECIPIENT_PK = 0xB0B;
    uint256 internal constant OPERATOR_PK = 0x0DE;

    MyConfidentialToken internal token;

    address internal holder;
    address internal recipient;
    address internal operator;

    function setUp() public override {
        super.setUp();

        holder = vm.addr(HOLDER_PK);
        recipient = vm.addr(RECIPIENT_PK);
        operator = vm.addr(OPERATOR_PK);

        token = new MyConfidentialToken(/* args */);
        _mintForTest(holder, 1000);
    }
}
```

If the project has no public mint/burn interface, create a test-only harness under `test/` to expose the necessary hooks. Do not put that harness in production contracts.

## Recommended Helpers

```solidity
function _encryptAmount(uint256 pk, uint64 amount)
    internal
    returns (externalEuint64 encryptedAmount, bytes memory proof)
{
    return encryptUint64(amount, vm.addr(pk), address(token));
}

function _decryptHandle(uint256 pk, euint64 value) internal returns (uint64) {
    address user = vm.addr(pk);
    bytes memory sig = signUserDecrypt(pk, address(token));
    return uint64(userDecrypt(euint64.unwrap(value), user, address(token), sig));
}

function _decryptBalance(uint256 pk, address account) internal returns (uint64) {
    return _decryptHandle(pk, token.confidentialBalanceOf(account));
}

function _decryptTotalSupply(uint256 pk) internal returns (uint64) {
    return _decryptHandle(pk, token.confidentialTotalSupply());
}

function callUserDecrypt(bytes32 handle, address user, address contractAddress, bytes memory sig)
    external
    returns (uint256)
{
    return userDecrypt(handle, user, contractAddress, sig);
}
```

`callUserDecrypt` is only for `expectRevert`, because reverts from internal calls are awkward to capture directly.

The two encryption-input bindings most often get written incorrectly:

- The target in `encryptUint64(amount, signer, address(token))` must be the token contract that ultimately calls `FHE.fromExternal`.
- `signer` should match the caller/operator that actually initiates that token operation.

## Balance and ACL

Positive path: the balance owner can decrypt.

```solidity
function test_holderCanDecryptBalance() public {
    assertEq(_decryptBalance(HOLDER_PK, holder), 1000);
}
```

Negative path: a third party cannot decrypt the holder's balance handle.

```solidity
function test_thirdPartyCannotDecryptBalance() public {
    uint256 thirdPk = 0xCAFE;
    address third = vm.addr(thirdPk);
    euint64 balance = token.confidentialBalanceOf(holder);
    bytes memory sig = signUserDecrypt(thirdPk, address(token));

    vm.expectRevert(
        abi.encodeWithSelector(
            FhevmTest.UserNotAuthorizedForDecrypt.selector,
            euint64.unwrap(balance),
            third
        )
    );
    this.callUserDecrypt(euint64.unwrap(balance), third, address(token), sig);
}
```

Test `confidentialTotalSupply()` the same way: write positive cases for users who should decrypt and negative cases for users who should not.

## Mint / Burn

Mint and burn use the same encrypted-input and decrypt helpers:

```solidity
function test_mintUpdatesBalanceAndSupply() public {
    (externalEuint64 amount, bytes memory proof) = _encryptAmount(HOLDER_PK, 250);

    vm.prank(holder);
    token.mintConfidential(recipient, amount, proof); // replace with the project's mint interface

    assertEq(_decryptBalance(RECIPIENT_PK, recipient), 250);
    assertEq(_decryptTotalSupply(HOLDER_PK), 1250);
}
```

Also cover non-minter calls, zero address, insufficient burn balance, and whether balance/total supply after burn match the intended design.

## Transfer

External-input version:

```solidity
function test_transferEncryptedAmount() public {
    (externalEuint64 amount, bytes memory proof) = _encryptAmount(HOLDER_PK, 400);

    vm.prank(holder);
    token.confidentialTransfer(recipient, amount, proof);

    assertEq(_decryptBalance(HOLDER_PK, holder), 600);
    assertEq(_decryptBalance(RECIPIENT_PK, recipient), 400);
}
```

Cover at least these failure paths too:

- Insufficient balance: the transaction may not revert, but balances should remain unchanged.
- Zero receiver: revert according to the project/standard expectation.
- Signer and caller mismatch.
- Proof target is not the token contract.

## Operator / transferFrom

The operator's encrypted input should be signed by the operator, because the operator is the actual caller of `confidentialTransferFrom`.

```solidity
function test_operatorTransferFrom() public {
    vm.prank(holder);
    token.setOperator(operator, uint48(block.timestamp + 1 days));

    (externalEuint64 amount, bytes memory proof) = _encryptAmount(OPERATOR_PK, 250);

    vm.prank(operator);
    token.confidentialTransferFrom(holder, recipient, amount, proof);

    assertEq(_decryptBalance(HOLDER_PK, holder), 750);
    assertEq(_decryptBalance(RECIPIENT_PK, recipient), 250);
}
```

Also test:

- Unauthorized operator calls fail.
- After `vm.warp(expiry + 1)`, `isOperator(holder, operator)` is false and `transferFrom` fails.
- When the operator uses an existing handle, the holder must already have authorized that handle for the operator.

## Existing Handle

If the token supports `confidentialTransfer(to, euint64 amount)`, the core test is ACL, not proof:

```solidity
function test_transferExistingHandle() public {
    euint64 holderBalance = token.confidentialBalanceOf(holder);

    vm.prank(holder);
    token.confidentialTransfer(recipient, holderBalance);

    assertEq(_decryptBalance(RECIPIENT_PK, recipient), 1000);
}
```

Negative case: `holder` cannot use `recipient`'s balance handle as the amount to transfer away; operators also cannot bypass handle ACL.

## transferAndCall

If the project supports callbacks, keep a focused set of behavior tests:

- Callback succeeds: transfer is applied and the receiving contract gets the expected data.
- Callback returns failure or rejects receipt: revert or refund according to the implementation.
- Callback itself reverts: the error is propagated.
- Recipient is an EOA: handle according to the standard/project expectation.
- When the operator uses `confidentialTransferFromAndCall`, operator authorization and input-signer binding still apply.

## Event Amount

If the transfer event exposes an encrypted amount handle, use `vm.recordLogs()` to capture the event, then user-decrypt the `euint64` handle from the event:

```solidity
vm.recordLogs();
_transferForTest(holder, recipient, 400);

euint64 eventAmount = _amountHandleFromTransferLog(vm.getRecordedLogs());
assertEq(_decryptHandle(HOLDER_PK, eventAmount), 400);
assertEq(_decryptHandle(RECIPIENT_PK, eventAmount), 400);
```

Third-party decryption of the event amount should fail. Write the event parsing helper against the project's event structure; the skill does not need to prescribe that implementation.

## Disclose / Public Reveal

If the project supports public reveal of an encrypted amount, focus on proof encoding:

```solidity
function test_discloseBalance() public {
    euint64 balance = token.confidentialBalanceOf(holder);

    vm.prank(holder);
    token.requestDiscloseEncryptedAmount(balance);

    uint64 clear = _decryptHandle(HOLDER_PK, balance);
    bytes memory proof = buildDecryptionProof(euint64.unwrap(balance), abi.encode(clear));

    vm.prank(holder);
    token.discloseEncryptedAmount(balance, clear, proof);
}
```

The proof generated by `publicDecrypt(handles)` matches `abi.encode(uint256[] cleartexts)` by default. If the contract verifies `abi.encode(uint64 clear)` or another custom ABI, use `buildDecryptionProof(handle, abi.encode(...))`.

At minimum, test these failure paths: unauthorized request, empty proof, wrong handle, wrong cleartext, and repeated disclose.

## Wrapper

Only needed when testing `IERC7984ERC20Wrapper`:

```solidity
dealConfidential(wrapper, user, amount);
```

It prepares a wrapper confidential balance before the test. Standard ERC7984 tokens do not need this helper.

## Common Errors

- The input proof target is not the token contract.
- The encrypted input signer does not match `msg.sender`/operator.
- Only transient ACL was granted, so cross-transaction `userDecrypt` fails.
- Expected persistent ACL was not configured for balance, event amount, or total supply.
- Operator authorization has expired, or the test did not reauthorize after `vm.warp`.
- Disclose uses the wrong proof ABI encoding.
- Assuming insufficient balance always reverts; many confidential tokens fail closed, so assert unchanged balances.
