# Testing ERC7984 Confidential Tokens

This document focuses on testing ERC7984 behavior with Hardhat/`@fhevm/hardhat-plugin`. The exact token, mint, burn, wrapper, and disclose interfaces depend on the project implementation.

## Test Surface

| Behavior | Hardhat FHEVM API | Focus |
| --- | --- | --- |
| balance decrypt | `userDecryptEuint(FhevmType.euint64, ...)` | owner can decrypt; third parties cannot |
| mint / transfer input | `createEncryptedInput(...).add64(...).encrypt()` | correct target/user binding |
| operator / transferFrom | encrypted input bound to the operator | unauthorized fails, authorized succeeds, expired fails |
| existing handle transfer | user decrypt + ACL | cannot spend someone else's handle |
| event amount | logs + user decrypt | sender/recipient can decrypt event amounts |
| disclose / public reveal | public decrypt or project callback helper | proof/signature encoding matches the implementation |
| wrapper | project wrapper helper | prepare balances only for wrapper tests |

## Basic Helpers

```ts
async function encryptAmount(tokenAddress: string, signer: HardhatEthersSigner, amount: bigint) {
  return fhevm
    .createEncryptedInput(tokenAddress, signer.address)
    .add64(amount)
    .encrypt();
}

async function decryptBalance(
  token: MyConfidentialToken,
  tokenAddress: string,
  account: string,
  signer: HardhatEthersSigner,
) {
  const handle = await token.confidentialBalanceOf(account);
  return fhevm.userDecryptEuint(FhevmType.euint64, handle, tokenAddress, signer);
}
```

## transfer

```ts
const encrypted = await encryptAmount(tokenAddress, alice, 100n);

await token
  .connect(alice)
  ["confidentialTransfer(address,bytes32,bytes)"](
    bob.address,
    encrypted.handles[0],
    encrypted.inputProof,
  );

expect(await decryptBalance(token, tokenAddress, bob.address, bob)).to.eq(100n);
```

Failure paths:

- Calling the current token after `createEncryptedInput(otherToken, alice.address)` should fail.
- Calling as Alice after `createEncryptedInput(token, bob.address)` should fail.
- Insufficient balance should revert or fail closed according to the implementation; it must not turn the balance into a wrapping result.

## operator

When the operator is the caller, the encrypted input user should be the operator:

```ts
await token.connect(alice).setOperator(operator.address, expiry);

const encrypted = await encryptAmount(tokenAddress, operator, 50n);
await token
  .connect(operator)
  ["confidentialTransferFrom(address,address,bytes32,bytes)"](
    alice.address,
    bob.address,
    encrypted.handles[0],
    encrypted.inputProof,
  );
```

Also test unauthorized and expired cases:

```ts
await expect(token.connect(operator).confidentialTransferFrom(...)).to.be.reverted;
```

## event amount

If the event exposes an encrypted amount handle:

```ts
const tx = await token.connect(alice)["confidentialTransfer(address,bytes32,bytes)"](...);
const receipt = await tx.wait();
const eventAmountHandle = extractAmountHandle(receipt!.logs);

expect(await fhevm.userDecryptEuint(FhevmType.euint64, eventAmountHandle, tokenAddress, alice)).to.eq(100n);
expect(await fhevm.userDecryptEuint(FhevmType.euint64, eventAmountHandle, tokenAddress, bob)).to.eq(100n);
```

Third-party decrypt should fail.

## disclose

If the token supports publicly revealing an amount:

- test that only authorized users can request it.
- test that finalize fails when there was no request.
- test wrong handle, wrong cleartext, and duplicate finalize failures.
- if on-chain `FHE.checkSignatures` is required, generate signatures/proofs with the project's callback helper. Do not use user decrypt as public proof.

Regular token tests do not need wrapper helpers. Prepare the underlying token and wrap only in `IERC7984ERC20Wrapper` tests.
