# Development Pattern: ERC7984 Confidential Token Integration

## What ERC7984 Is

ERC7984 is the base confidential fungible token implementation in OpenZeppelin confidential contracts. It uses FHEVM encrypted balances and encrypted transfer amounts to implement private tokens. It fits projects that need to hide balances and transfer amounts while preserving ERC-style token workflows.

Official repositories:

- `https://github.com/OpenZeppelin/openzeppelin-confidential-contracts`
- Zama examples usually live under docs/examples/openzeppelin in `https://github.com/zama-ai/fhevm`.

## Basic Dependencies

Contracts usually need:

```solidity
import {Ownable2Step, Ownable} from "@openzeppelin/contracts/access/Ownable2Step.sol";
import {ERC7984} from "@openzeppelin/confidential-contracts/token/ERC7984/ERC7984.sol";
import {FHE, euint64, externalEuint64} from "@fhevm/solidity/lib/FHE.sol";
import {ZamaEthereumConfig} from "@fhevm/solidity/config/ZamaConfig.sol";
```

Exact import paths depend on the current OpenZeppelin confidential contracts version.

## Minimal Token Example

```solidity
// SPDX-License-Identifier: BSD-3-Clause-Clear
pragma solidity ^0.8.27;

import {Ownable2Step, Ownable} from "@openzeppelin/contracts/access/Ownable2Step.sol";
import {ERC7984} from "@openzeppelin/confidential-contracts/token/ERC7984/ERC7984.sol";
import {FHE, euint64} from "@fhevm/solidity/lib/FHE.sol";
import {ZamaEthereumConfig} from "@fhevm/solidity/config/ZamaConfig.sol";

contract ConfidentialToken is ZamaEthereumConfig, ERC7984, Ownable2Step {
    constructor(
        address owner,
        uint64 initialAmount,
        string memory name_,
        string memory symbol_,
        string memory contractURI_
    ) ERC7984(name_, symbol_, contractURI_) Ownable(owner) {
        euint64 encryptedAmount = FHE.asEuint64(initialAmount);
        _mint(owner, encryptedAmount);
    }
}
```

This example initializes supply with plaintext `initialAmount`, which is suitable for public tokenomics. If the initial supply also needs to remain private, use confidential mint instead.

## Visible Mint and Confidential Mint

Visible mint: the amount is public and suits public issuance plans.

```solidity
function mintVisible(address to, uint64 amount) external onlyOwner {
    euint64 encryptedAmount = FHE.asEuint64(amount);
    _mint(to, encryptedAmount);
}
```

Confidential mint: the amount is encrypted by the minter and suits hidden issuance or allocation amounts.

```solidity
function mintConfidential(
    address to,
    externalEuint64 encryptedAmount,
    bytes calldata inputProof
) external onlyOwner {
    euint64 amount = FHE.fromExternal(encryptedAmount, inputProof);
    _mint(to, amount);
}
```

Check the current OpenZeppelin confidential contracts version for the exact `_mint` signature.

## Transfer Interface

ERC7984 confidential transfers usually receive a recipient, encrypted amount handle, and input proof. In the ABI, `externalEuint64` may appear as `bytes32`, but semantically it is still an external encrypted input verified with a proof.

```solidity
function transferWithEncryptedAmount(
    address recipient,
    externalEuint64 encryptedAmount,
    bytes calldata inputProof
) external {
    euint64 amount = FHE.fromExternal(encryptedAmount, inputProof);
    FHE.allowTransient(amount, address(confidentialToken));
    confidentialToken.confidentialTransfer(recipient, amount);
}
```

## Cross-contract Calls

If the current contract holds an `euint64 amount` and needs to hand it to an ERC7984 token contract for a transfer, transiently authorize the token:

```solidity
FHE.allowTransient(amount, address(confidentialToken));
confidentialToken.confidentialTransfer(recipient, amount);
```

Auctions, vesting, AMMs, and wrappers all use this pattern.

## Balance Reads

```ts
const balanceHandle = await token.confidentialBalanceOf(user.address);
```

Whether user decrypt succeeds depends on the ERC7984 implementation and the current user's ACL permissions. Tests should not only assert that a handle exists; critical paths must cover both successful user decrypt and unauthorized failure.

## Vesting Pattern

A confidential vesting wallet usually keeps time conditions public and amounts encrypted:

```solidity
function releasable(address token) public returns (euint64) {
    euint128 vested = vestedAmount(token, uint48(block.timestamp));
    euint128 releasedAmount = released(token);
    ebool ok = FHE.ge(vested, releasedAmount);
    return FHE.select(ok, FHE.asEuint64(FHE.sub(vested, releasedAmount)), FHE.asEuint64(0));
}
```

When releasing, transiently authorize the releasable amount to the token:

```solidity
euint64 amount = releasable(token);
FHE.allowTransient(amount, token);
euint64 sent = IERC7984(token).confidentialTransfer(owner(), amount);
```

## Integration Notes

- ERC7984 usually uses `euint64` for amount width. Do not change to another width without confirmation.
- Initial minting with a plaintext amount converted to an encrypted amount reveals initial supply. If the business requires full privacy, design encrypted mint.
- The `confidentialTransfer` amount must be encrypted by the sender and the transaction must be sent by that sender.
- Check zero address, zero balance, insufficient balance, and similar error paths.
- When integrating with ordinary ERC20 wrappers, DEXes, vesting, or bridges, be explicit about which amounts may be public and which must remain private.
- Check whether `allowTransient` is needed before using an encrypted amount across contracts.
- Wrapper/unwrap flows expose some information to the ERC20 world, so document the privacy assumptions first.

## Common Extensions

- Owner-only mint: if the mint amount is a public management operation, use `FHE.asEuint64`; if the mint amount should also remain private, use encrypted input.
- Burn: verify the encrypted amount just like transfer, and handle insufficient balance.
- Wrapper: when converting between ERC20 and ERC7984, public wrap/unwrap amounts affect the privacy assumptions.
- Compliance: compliance reads can be designed with user decrypt delegation or selective disclosure, but they must be explicitly authorized and recorded.
- Sealed-bid auction: collect encrypted bids with ERC7984, update highest price and encrypted winner with `FHE.select`, then public decrypt the winner after the auction ends.
